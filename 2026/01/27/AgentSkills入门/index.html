Agent Skills 入门

1. 核心定义

Agent 即智能体，Skills 对应技能的概念。在 Agent Skills 体系中，Skills 模块会将特定领域的知识与工作流进行封装，智能体可通过调用这些 Skills 组件，高效完成特定业务任务。

单个 Skill 以独立文件夹的形式存在，其内部必须包含一份 SKILL.md 格式文件，用于指导 Claude 在特定业务场景下按照预设逻辑执行操作；同时可按需搭配脚本（scripts）、参考文档（references）、素材资源（assets）等附属文件。

简言之，传统模式下我们依靠提示词（Prompt）指导 AI 完成任务，而 Agent Skills 则能将提示词与各类配套资源整合封装，形成可复用、可共享的标准化技能包。

Agent Skills 的核心机制在于渐进式披露，该过程通常分为三个递进步骤依次执行：

  1. AI 首先读取所有技能包内 SKILL.md 文件的元数据（包含 name 与 description 字段），以此判断当前技能包与待执行任务的相关性。
  2. 若判定技能包与任务相关，AI 会自动读取 SKILL.md 的正文内容，从中获取任务执行的详细指导规则与流程要求。
  3. AI 仅在任务执行需要时，才按需读取脚本、示例等附属文件，或调用工具执行对应脚本逻辑。

SKILL.md 相当于整个技能包的核心指挥文件，它明确界定了 AI 调用 scripts 目录下代码的时机，以及读取 references 目录下文件的场景。

3. 背景来源

早期开展 AI Agent 开发工作时，技术人员常会直接将各类 API 文档、业务规则、数据库结构等内容，全部嵌入系统提示词中。这种方式存在两个明显弊端：

  1. 提示词篇幅过长，易导致 AI 出现生成幻觉，或引发注意力衰减问题，影响任务执行效果。
  2. 能力逻辑固化于提示词内，无法实现灵活复用，难以适配多场景需求。

针对上述痛点，Claude 开发团队 Anthropic 主导提出了 Agent Skills 标准化方案，将 AI 执行任务所需的“能力组件”从对话上下文中剥离，实现了能力的标准化定义、外置化存储与模块化调用。我们常提及的、能够读取技能包并执行指定任务的工具，正是 Claude Code 这款 IDE；目前组内开发所使用的 Cursor 工具，同样支持该功能。需要说明的是，这类 IDE 直接调用 Agent Skills 相关功能通常为付费模式，不过也存在“开源框架 + 付费 API”的组合方案，具体技术选型需后续在所内进一步评估确定。

4. 核心价值

  • AI 本质上属于概率模型，输出结果存在不确定性。而 Agent Skills 依托硬性的代码约束（Scripts）与严谨的文档规范（SOP），可有效确保 AI 在处理关键业务任务时，输出结果具备确定性与可复现性。
  • 通过编写 SKILL.md 文件，能够将特定业务场景下的项目代码流程封装为标准化技能包，将业务场景中的隐性经验固化为显性化的文件资产。同时，该技能包的边际复制成本几乎为零，可快速在多场景落地复用。
  • SKILL.md 聚焦于定义标准化的业务逻辑，明确规定了任务的输入要求、输出标准以及中间需调用的脚本组件。无论底层大模型如何迭代更换，这套业务逻辑都具备独立性与可移植性，无需像以往那样，每更换一次大模型就重新调试适配提示词。
  • 对于 Cursor 来说，与始终生效的声明式 rules 相比，技能更适合用于动态上下文发现和过程式的“操作指南”类说明。这样既能让 Agent 拥有更高的灵活性，又能让上下文保持聚焦。

5. 概念解析

  • SKILL.md vs Prompt：Prompt 是 AI 任务的「一次性指令」，而 SKILL.md 是「可复用的标准化技能说明书」。SKILL.md 本质上是将 Prompt 的逻辑与外部资源打包，解决了长 Prompt 幻觉、不可复用的痛点。
  • SKILL.md vs MCP：MCP 是「模型与工具对话的通用语言」，SKILL.md 是「基于该语言编写的业务操作指南」。没有 MCP，SKILL.md 中定义的工具调用逻辑无法落地；没有 SKILL.md，MCP 仅能实现零散的工具调用，无法支撑标准化的业务技能。
  • SKILL.md vs RAG：RAG 是「解决信息缺失的技术手段」，SKILL.md 是「封装业务逻辑与资源的载体」。SKILL.md 可以把 RAG 所需的知识库、检索规则、执行脚本打包成技能包，让 RAG 能力从「单次检索」升级为「可复用的检索技能」。

Prompt 是 SKILL.md 的「前身」
MCP 是 SKILL.md 的「技术底座」
RAG 是 SKILL.md 的「能力子集」
